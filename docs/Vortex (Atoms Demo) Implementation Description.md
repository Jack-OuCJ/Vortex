# 1. 概述

这个 Demo 的目标不是机械复刻 Atoms 的全部能力，而是在有限时间内优先完成一条真实可运行的核心链路：用户输入需求后，系统能够创建项目、进入工作台、通过 AI 生成或修改代码、在浏览器内实时运行预览，并将项目文件和对话历史持久化保存。围绕这个目标，我选择了 Next.js App Router 作为前后端一体化框架，用 Supabase 承接鉴权和数据存储，用 WebContainer 在浏览器中提供类 IDE 的文件系统、命令执行和预览能力。

这次实现更关注“从想法到可运行页面”的端到端体验。用户从首页输入需求后，可以直接创建项目、进入工作台、看到 AI 流式分析与执行步骤、实时生成代码、在右侧预览运行结果，并在刷新后恢复项目文件和聊天上下文。整体上，这是一个偏产品原型但已经具备真实交互和持久化能力的 Atoms Demo。
# 2. 完成程度

## 2.1 已经实现的内容

当前版本已经完成了 Demo 的核心主流程，并且具备较强的可演示性。

首先，基础使用流程已经打通。用户可以通过邮箱 Magic Link 或 Google OAuth 登录，登录后从首页输入需求创建项目，系统会先创建项目记录，再跳转到工作台开始生成流程。首页同时提供模型选择、附件上传和历史项目入口，能支撑一个完整的初次使用和回访流程。

其次，AI 工作台已经具备真实交互能力，而不是静态展示。用户在工作台输入需求后，后端会通过流式接口返回任务分析、执行步骤、工具调用结果和最终交付内容。前端会实时展示聊天内容、工作流程、文件树和预览界面，用户可以感知“AI 正在做什么”，而不是等待一个黑盒式结果。

再次，代码生成不是纯文本输出，而是带有文件级落地和运行预览的闭环。AI 会基于当前项目上下文进入开发模式，读取文件、写入文件、执行命令，并将变更同步到浏览器中的运行环境。右侧预览区域会随着文件同步更新，用户可以直接看到生成结果是否可运行。

此外，编辑器和项目管理也已经形成可用体验。当前版本支持文件树浏览、Monaco 编辑器手动改代码、保存文件、预览刷新、项目重命名、删除项目，以及历史会话恢复。用户既可以让 AI 生成，也可以在 AI 生成后自己接手微调，这一点对 Demo 的“可用性”比单纯生成一段代码更重要。

最后，数据持久化已经落地。项目、项目文件、聊天会话、聊天消息都存储在 Supabase 中，页面刷新后可以恢复最近活跃会话。

## 2.2 当前尚未完全覆盖的部分

虽然核心链路已经完成，但当前版本仍然有一些明确的边界。

第一，当前版本已经具备可对外访问的在线站点，能够满足 Demo 的在线体验与验证需求。但从产品能力上看，现阶段更偏“可访问的完整原型”而非“完善的公开分享平台”：站点访问和主流程体验已经可用，后续仍可以继续补强面向单个项目的公开展示页、分享链接管理和更完整的对外作品分发能力。

第二，产品界面已经呈现“AI 团队”概念，但后端执行目前以单条工程师执行链路为主，重点放在从需求到文件修改的稳定闭环上。多角色协作的产品表达已经具备，但真正强约束的多 Agent 串联调度还可以继续深化。

第三，当前版本更强调快速成型和关键能力闭环，因此在失败恢复、版本快照、分享协作、执行审计、命令沙箱治理等生产级能力上仍有扩展空间。
# 3. 实现思路与关键取舍
## 3.1 登录模块

登录模块基于 Supabase Auth 实现，支持邮箱 Magic Link 和 Google OAuth 两种方式。

这个选择的核心原因是：在有限时间里，鉴权并不是 Demo 的差异化重点，但它又直接影响项目持久化、用户隔离和在线可体验性。使用 Supabase Auth 可以快速拿到一套可运行的登录能力，同时天然与后续数据库 RLS 策略打通。

在实现上，我把登录流程设计为轻量、低阻力的进入方式。邮箱登录适合最低门槛的体验，Google OAuth 更适合快速注册和保留头像、用户名等资料。登录成功后，用户会回到首页继续进入项目创建流程，不需要再额外理解复杂的账户系统。

## 3.2 AI 额度计算
当前版本保留了一个轻量的 AI 额度机制，用于体现产品层面的资源约束，而不是无限制调用模型。用户资料中存储了当前额度和最大额度，进入 AI 生成链路前，后端会先检查额度是否充足；若额度不足，会直接中断调用并返回明确提示。生成完成后，再通过数据库函数扣减额度。

我在这里的取舍是没有做复杂计费模型，而是采用“固定额度 + 每轮扣减”的轻量方案。这样做的好处是实现简单、产品语义明确，它不是精确的商业计费系统，但足够支撑一个真实可体验的演示场景。
## 3.3 智能体设计

在智能体设计上，我刻意没有把系统做成“只会回复一段结果文本”的聊天机器人，而是把它设计为一个带执行闭环的工程师代理。当前版本会先判断用户输入属于普通问答、从零构建、继续开发、局部修改还是问题修复；如果判断为开发任务，则进入文件级执行流程。

这套设计的核心价值是让 AI 不只是“说自己会做什么”，而是“真正对项目文件做什么”。后端会把当前项目文件、热点文件、文件树摘要和最近对话上下文提供给模型，让模型在这些约束下输出结构化决策，再驱动文件读写、目录操作和命令执行。

从产品表达上，界面保留了团队角色设定，例如 Emma、Bob、Alex 等，以贴近 Atoms 的 Team Mode 感受；但在工程实现上，我优先确保工程师执行链路可用，把多角色更多用于工作流表达和后续扩展接口，而不是在首版里追求复杂的多 Agent 状态机。这个取舍能明显降低系统不稳定因素，让 Demo 先站住核心体验。
## 3.4 交互流程设计

整个交互流程是围绕“让用户持续感受到系统正在工作”来设计的。首页负责项目入口，工作台负责承接完整开发过程。用户从首页输入需求后，不会停留在一个单独的提交页，而是直接进入工作台，这样可以立即看到对话、步骤流、文件树和预览区域，产品反馈更加即时。

在工作台内部，我把交互拆成两个同步区域：左侧对话与步骤流、右侧运行预览。AI 的流式响应会被拆分为不同事件类型，前端分别映射为聊天消息、步骤状态、工具执行和预览更新。这种设计是为了尽量接近真实 AI IDE 的体验，让用户知道系统是如何一步步把需求变成应用的。

相比只返回一个最终答案，这种流程设计更符合 Atoms 这类产品的价值主张，也更能体现 Demo 的完整度。
## 3.5 编辑器设计与运行时方案取舍

编辑器和运行时是这次实现里最关键的技术选择之一。我实际评估过 Sandpack、WebContainer 和 E2B 三种方案，最终首版选择了 WebContainer。

首先，Sandpack 的优点是接入轻、展示快，适合快速做一个“能看见页面结果”的前端沙箱。但它更偏展示型运行环境，对于“让智能体像操作一个小型开发环境一样去读文件、写文件、建目录、执行命令、读取进程输出”这类需求，能力边界会比较明显。如果用 Sandpack 做首版，我可以更快做出一个页面预览器，但很难自然承接当前这条工具级 AI 执行链路。

其次，E2B 的方向很有吸引力，尤其适合后续做更强隔离、更重任务、更接近云端开发环境的执行模型。如果项目进入下一阶段，E2B 可以承担云端沙箱、远程执行、多人协作、长生命周期任务等更重的能力。但是对于的 Demo，E2B 的接入、远程环境编排、资源治理和链路调试成本都更高，不利于在有限时间里尽快做出一个稳定、可演示、端到端闭环的版本。

最终选择 WebContainer，核心原因有三点。
* 第一，WebContainer 足够接近真实开发环境，能够在浏览器里提供文件系统、命令执行和本地预览，这使得“AI 修改代码并立即运行”的体验非常顺滑。
* 第二，WebContainer 对当前智能体操作模型非常友好。因为我可以把 AI 的行为抽象成一组明确的工具，例如读取文件、写入文件、列目录、创建目录、执行命令、读取进程输出、终止进程，然后通过前后端桥接把这些工具映射到浏览器中的 WebContainer 实例上。这使得 AI 不只是生成代码文本，而是在操作一个真实的运行时环境。
* 第三，WebContainer 更适合这次 Demo 的目标，也就是“快速成型”。它让我可以在不引入额外云端沙箱基础设施的前提下，把产品体验先做完整：创建项目、生成文件、安装依赖、启动预览、展示结果。对一版原型来说，开发效率和可演示稳定性是第一优先级，WebContainer 在这方面是最平衡的选择。

因此，这次的判断不是“WebContainer 永远优于 E2B”，而是“在当前阶段，WebContainer 是最快让产品原型成立的方案，E2B 是后续做云端运行时与更强隔离能力的自然迭代方向”。这也是我在技术路线上的核心取舍。
## 3.6 数据持久化设计

数据持久化采用 Supabase，主要覆盖四类核心对象：用户资料、项目、项目文件、聊天会话与聊天消息。之所以优先把这部分做完整，是因为 Atoms 这类产品如果没有持久化，就会退化成一次性生成器，无法支撑连续迭代。

在具体设计上，项目与文件是分离存储的，项目负责元信息，文件负责真实代码内容。这样做的好处是更适合增量同步，也方便在 AI 生成和用户手动编辑之间做冲突处理。聊天数据则按 session 和 message 分层存储，便于恢复最近活跃会话和展示历史记录。

另一个关键取舍是：文件持久化不能只依赖前端防抖保存。当前版本中，AI 生成完成后，后端会在同一轮流程中把变更文件增量回写到数据库，并把最新时间戳返回前端，以减少刷新后丢内容或者本地状态漂移的问题。这个决定提升了系统一致性，也让 Demo 更接近可用产品，而不是单纯前端内存态演示。

# 4. 扩展功能和优先级判断

如果继续投入时间，我会优先做下面几个方向。

第一优先级是把当前“单条工程师执行链路”扩展为更清晰的多智能体编排，例如把需求澄清、规划、实现、修复拆成独立节点，并引入更强的结构化状态流。这样可以进一步贴近 Atoms 的 Team Mode，也能让步骤流展示更真实，而不只是 UI 层面的角色包装。

第二优先级是把运行时从“浏览器内快速成型”进一步扩展到“云端隔离执行”。这时 E2B 会成为很自然的下一步。WebContainer 适合当前原型期，但如果后续要支持更重的执行任务、更强隔离、长生命周期进程、多人共享和稳定的公开运行环境，那么 E2B 会比浏览器侧容器更有扩展潜力。

第三优先级是完善产品细节，包括版本快照、回滚、失败重试、更细粒度的权限治理、执行日志面板、分享协作以及更多模板化起步能力。它们不会改变主链路，但会显著提升产品质感和真实可用性。

# 5. 功能展望
当前的AI IDE仍然需要自然语言去驱动Agent进行应用生成。
后续我想让应用设计的过程更加得智能化和有交互感，可以让Agent预先和用户交流选型，选定合适的网页ui设计之后再进行代码生成。这样可以让用户在设计阶段就有更多的参与感，而不是完全把需求丢给AI去理解和实现。

总体来说，这次实现的策略是先用 WebContainer 快速把“AI 生成应用并立即看到结果”的核心体验做出来，让 Demo 先具备真实交互和持久化能力；在此基础上，再逐步把系统往更强的多智能体编排和云端沙箱方向演进，而 E2B 正是后续迭代中非常合适的扩展路线。
